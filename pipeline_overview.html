长期对话数据生成 Pipeline

本系统调用兼容 OpenAI API 的模型，根据固定 CaseSpec 生成按时间排列的长期用户—助手对话，用于构建情绪记忆触发评测数据集。

项目概览

本项目根据固定 CaseSpec（案例规格）调用兼容 OpenAI API 的模型，生成按时间排列的长期用户—助手对话。每个角色产出：

  • 10 个 session：按时间排列的历史对话，每个 session 包含 4–6 个 round（user turn + assistant turn）；
  • 6 个 eval sample：用于评测的当前输入与 gold 答案；
  • 1 份 QA 报告：自动结构校验结果；
  • 1 份 benchmark JSON：最终输出数据集。

所有 LLM 输出均使用结构化 JSON Schema约束，并通过 Pydantic 模型校验；每个步骤失败时最多重试 3 次。

核心任务：模型需判断 trigger_label（triggered / not_triggered / insufficient_evidence），即当前输入中的 cue 是否触发用户的历史情绪记忆。触发成立需同时满足：① cue 出现在当前输入中；② 历史中存在 cue 与个人情绪事件的关联证据；③ 该历史记忆能够解释用户当前的情绪或行为反应。

Pipeline 整体流程

1
加载 CaseSpec
从 cases/case_a.json 读取角色身份、核心情绪事件、cue 规格、10 个 session outlines 和 6 个 eval outlines。这是后续生成的事实唯一来源。
2
分批生成 SessionPlan（SessionPlanner）
按 planner_batch_size 分批生成轻量计划，每批将已生成计划传给下一批以维持严格时间顺序，避免单次输出长度限制。产出 session_plans.json 与 session_plans.txt。
3
逐 Session 生成对话（SessionWriter → Verifier → Reviser）
每个 session 依次执行：① SessionWriter 生成完整对话；② _structural_verdict 做结构校验（turn 数、ID、speaker 顺序）；③ SessionVerifier 做语义校验；④ 若需修订，SessionReviser 做最小修改。最多循环 max_revision_cycles 次。
4
自然度检查（NaturalnessChecker）
检查对话风格、语气是否自然；如需修订，调用 Reviser 仅修改指定表达，不得改变事件、时间、人物、cue。
5
Checkpoint 持久化
每完成一个 session 立即写入 checkpoint_sessions.json，便于中断后从未完成处继续。重新运行时会校验并复用已有计划与合法 checkpoint。
6
QA 校验（validate_sessions）
非 LLM 校验：session ID 顺序、严格时间递增、turn 数量与交替规则、speaker 唯一性。产出 qa_report.json。
7
输出 Benchmark JSON
汇总 dataset_id、character_profile、dialogues、eval_samples，写入 benchmark.json。本轮 run_eval=false，eval_samples 为空数组。
8
Eval 生成（可选，当前关闭）
EvalGenerator 生成 3 个候选 → EvalVerifier 选择最佳或全部拒绝 → GoldFinalizer 做确定性整理（校验 turn/cue 存在性、随机排序 cue options、分配 sample ID）。后续可在 pipeline 中启用。

核心组件功能

SessionPlanner LLM

prompts/session_planner.txt
职责：将固定 CaseSpec 扩展为轻量 SessionPlan
  • 宏观控制：指定每个 session 的主题、日期、story beat；
  • 保持固定故事功能不变，不得改变核心事件；
  • 支持分批生成，把已生成计划传给下一批维持时间顺序。
输入：CaseSpec + min/max rounds + prior_plans
输出：SessionPlanList（session_id, date, topic, story_beat, round_count）

SessionWriter LLM

prompts/session_writer.txt
职责：一次生成完整 session 对话
  • 根据 plan 和 recent_history 生成全部 turns；
  • 严格遵循 user/assistant 交替、round 结构、turn ID 命名；
  • 保持角色风格与故事 beat 一致。
输入：CaseSpec + SessionPlan + 最近 N 个 session
输出：Session（含 turns、summary）

SessionVerifier LLM

prompts/session_verifier.txt
职责：只做检查，不改文本
  • assistant 是否知道了未披露信息（overclaim）；
  • 人物身份是否冲突；
  • 是否形成机械问答、是否重复历史；
  • user 是否有可用 evidence turn；
  • session 是否像自然对话。
输入：CaseSpec + plan + Session + recent_history
输出：VerificationResult（pass/revise + issues 列表）

SessionReviser LLM

prompts/session_reviser.txt
职责：根据具体问题做最小修改
  • 仅修改 verifier/naturalness 指出的相关 turn；
  • 不重写整个 session；
  • 保留事件、时间、人物、cue 不变。
输入：CaseSpec + plan + Session + VerificationResult
输出：修订后的 Session

NaturalnessChecker LLM

prompts/naturalness_checker.txt
职责：检查对话风格与自然度
  • 基于 character_profile.conversation_style；
  • 检查语气、表达是否自然；
  • 只修改指定表达，不改事实。
输入：conversation_style + Session
输出：VerificationResult（pass/revise）

EvalGenerator LLM · Eval

prompts/eval_generator.txt
职责：根据覆盖目标生成 eval 候选
  • 按 eval outline 的 target_label/emotion/history_cutoff 出题；
  • 从历史中寻找可用 evidence；
  • 生成当前文本/图文输入、cue options、预期 gold；
  • 一次生成 3 个候选。
输入：CaseSpec + visible_sessions + EvalOutline
输出：EvalCandidateList（3 个候选）

EvalVerifier LLM · Eval

prompts/eval_verifier.txt
职责：检查候选并选择最佳
  • 不生成新题，不大幅重写；
  • 检查每个候选是否符合覆盖目标；
  • 选择最佳候选，或全部拒绝要求重试。
输入：CaseSpec + sessions + outline + candidates
输出：EvalSelection（reject_all + selected_index + issues）

GoldFinalizer 非 LLM

职责：确定性整理最终 eval JSON
  • 校验 evidence turn_id 是否存在且 speaker=user；
  • 校验 cue_id 是否存在于 cue_options；
  • 随机排序 cue options（避免位置偏见）；
  • 根据标签强制修正空值规则；
  • 分配 sample_id，删除调试字段。
输入：CaseSpec + sessions + EvalCandidate + 序号
输出：EvalSample

LLMClient 基础设施

职责：统一 LLM 调用客户端
  • 结构化输出（json_schema response_format）；
  • 按组件配置 model/temperature/max_tokens；
  • 从环境变量读取 api_key 与 base_url（自动补 /v1）；
  • 支持 openai_sdk 与 powershell 两种 transport；
  • 失败重试（最多 max_retries 次），记录 CallRecord。
输入：component 名 + payload + 输出模型类
输出：Pydantic 模型实例 + records 调用记录

validate_sessions 非 LLM

职责：QA 结构校验
  • session_ids_and_order：与 case outline 顺序一致；
  • strict_chronology：date 严格递增；
  • turn_structure：8–12 turns、user/assistant 交替、ID 唯一；
  • schema_validation：Pydantic 校验通过。
输入：dataset_id + CaseSpec + sessions + call_records
输出：QAReport（passed + checks + errors）

Gold 一致性规则（model_validator 强制）：
  • triggered：cue_id ≠ none，evidence 非空；
  • not_triggered / insufficient_evidence：cue_id = none，evidence 为空；
  • evidence 只能引用真实存在的 user turn。

输出目录结构

outputs/case_a/ ├── session_plans.json # 10 个 SessionPlan（结构化） ├── session_plans.txt # 人工检查用的纯文本计划 ├── checkpoint_sessions.json # 已完成 session 的增量 checkpoint ├── benchmark.json # 最终数据集（dialogues + eval_samples） └── qa_report.json # 结构校验报告 + 调用记录

运行方式

安装依赖并设置环境变量后运行：

# 设置 API 凭据 $env:openai_api_key = "your-key" $env:base_url = "https://your-gateway.com" # 可选，自动补 /v1 # 安装依赖 pip install -r requirements.txt # 运行 python run.py --case cases/case_a.json --output outputs/case_a
断点续跑：再次使用同一输出目录运行时，程序会校验并复用已有 session_plans.json 和合法的 session 前缀 checkpoint，从未完成的 session 继续。
